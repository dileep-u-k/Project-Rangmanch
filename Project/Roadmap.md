	•	raw concept

PHASE 0 — Strategic Grounding and Thesis Lock

⸻

0.1 Core objective

The purpose of Phase 0 is to ensure that the project is not built on confusion.

Before any design, code, prototype, or deck work begins, this phase locks:
	•	the exact meaning of the product,
	•	the problem it solves,
	•	why this category should exist,
	•	why this idea is different,
	•	what the brief really requires,
	•	and what kind of product you are actually building.

This phase transforms a broad idea into a thesis-grade product foundation.

⸻

0.2 Why this phase is essential

Most projects fail because they start too early with:
	•	features,
	•	UI,
	•	model prompts,
	•	or tech stack choices.

That produces:
	•	generic products,
	•	fuzzy differentiation,
	•	weak adoption logic,
	•	and decks that sound impressive but lack depth.

For RANGMANCH, Phase 0 is critical because the idea is ambitious and category-creating.
So the project must first answer:
	•	What is RANGMANCH, precisely?
	•	What is it not?
	•	What is the non-obvious insight?
	•	Why is this an AI companion product at all?
	•	Why is it entertainment-first?
	•	Why will users care?
	•	Why is it not a chatbot clone?

If this is not locked, later work will drift.

⸻

0.3 What must be done in Phase 0

A. Decode the case brief properly

You must rewrite the brief into founder-grade language.

The brief asks for:
	•	AI Companion product
	•	fun and entertainment based
	•	broadly adoptable
	•	unique
	•	end-to-end prototype
	•	TAM, adoption, growth, failure analysis  

But the deeper meaning is:
	•	invent a new category-worthy consumer entertainment product,
	•	prove that AI is core,
	•	and show that you can think like founder/product/strategy leadership.

This reframing must be written down.

⸻

B. Lock the one-line thesis

You must create one final thesis sentence.

Example structure:

RANGMANCH is an AI-native entertainment companion that transforms a user’s real-life moods, moments, and social context into personalized interactive episodes, social mini-dramas, and evolving story worlds.

This thesis must remain stable through all later phases.

⸻

C. Define category boundaries

You must explicitly define what the project is and what it is not.

Write:
	•	not therapist-first
	•	not chatbot clone
	•	not character AI copy
	•	not social feed app
	•	not casual game only
	•	not meme generator only
	•	not productivity assistant

And then define the actual category:
Companion-led personal entertainment operating system

This is essential for originality.

⸻

D. Lock the central metaphor

RANGMANCH’s metaphor must be fixed early:

Life is the stage.
The companion is the narrator, host, director, and chaos engine.
Daily reality becomes entertainment.

This metaphor will guide:
	•	naming
	•	UX
	•	output design
	•	deck storytelling
	•	companion voice
	•	artifact design

⸻

E. Define the strategic wedge

Before the project becomes too broad, lock the first wedge.

For the MVP, the wedge should be:
	•	Today’s Episode
	•	plus Squad Mode
	•	plus shareable artifact
	•	plus return hook

This wedge must be frozen in Phase 0.

⸻

F. Define project principles

Create a project principles sheet.

Recommended principles:
	•	entertainment first
	•	AI-native, not AI-coated
	•	short-session delight
	•	continuity over randomness
	•	social shareability
	•	memory without creepiness
	•	playful, not emotionally burdensome
	•	bold originality, but intuitive entry

These principles become decision filters later.

⸻

0.4 Deliverables of Phase 0

By the end of this phase, you should have these documents:

Deliverable 1 — Thesis Note

A short document containing:
	•	one-line project definition
	•	why it exists
	•	who it is for
	•	what category it defines
	•	what makes it original

Deliverable 2 — Project Boundary Note

Containing:
	•	what the product is
	•	what it is not
	•	what is in scope
	•	what is out of scope

Deliverable 3 — Product Principles Note

Containing the design and execution rules.

Deliverable 4 — MVP Wedge Freeze Note

Containing:
	•	flagship feature
	•	social wedge
	•	return mechanic
	•	what the first prototype must prove

⸻

0.5 Success criteria for Phase 0

Phase 0 is successful if:
	•	the product can be explained in one sentence,
	•	the project category is clear,
	•	the uniqueness logic is clear,
	•	the entertainment-first positioning is unambiguous,
	•	the MVP wedge is locked,
	•	and all future work can reference a stable thesis.

⸻

0.6 Common failure modes in Phase 0
	•	idea sounds exciting but too broad
	•	confusion between companion and chatbot
	•	unclear uniqueness
	•	trying to solve too many user problems
	•	accidental drift toward therapy/productivity
	•	no frozen wedge

⸻

0.7 Exit gate for Phase 0

Do not move to Phase 1 until you can answer clearly:
	1.	What is RANGMANCH in one sentence?
	2.	What exact product category is this?
	3.	What makes it different from existing AI companions?
	4.	What is the first strong wedge?
	5.	What should the MVP prove?

If these are not crisp, Phase 0 is incomplete.

PHASE 1 — Problem Research, Behavioral Insight, and Opportunity Discovery

⸻

1.1 Core objective

The purpose of Phase 1 is to build a serious, evidence-backed understanding of:
	•	user behavior,
	•	entertainment patterns,
	•	AI companion limitations,
	•	mobile social habits,
	•	and the opportunity space where RANGMANCH can win.

This phase gives the project research depth.

Without it, the product will sound imaginative but unsupported.

⸻

1.2 Why this phase matters

The brief explicitly values:
	•	originality
	•	creativity
	•	user empathy
	•	depth of research
	•	cross-sectoral thinking
	•	original insights uncovered  

So Phase 1 is not optional polish.
It is a direct scoring area.

This phase proves that the concept is not random invention.
It is built on:
	•	real behavior,
	•	real gaps,
	•	real usage patterns,
	•	and reasoned insight.

⸻

1.3 Research pillars of Phase 1

You should organize research into 6 pillars.

⸻

Pillar A — Entertainment behavior research

Study how users engage with:
	•	short-form entertainment
	•	casual entertainment apps
	•	social fun products
	•	shareable digital content
	•	micro-session engagement patterns
	•	boredom behavior
	•	post-work / post-class / late-night entertainment loops

Goal:
Understand when and why users seek light fun.

⸻

Pillar B — Companion product landscape research

Study existing AI companion products and adjacent categories:
	•	what they optimize for
	•	where they are chat-heavy
	•	where they are emotionally intense
	•	where they fail on social shareability
	•	where they fail on entertainment repeatability

Goal:
Define whitespace.

⸻

Pillar C — Social expression and group energy research

Study:
	•	friend-group banter
	•	inside-joke formation
	•	screenshot sharing behavior
	•	WhatsApp/DM forwarding behavior
	•	playful group interaction patterns
	•	social participation triggers

Goal:
Understand how RANGMANCH can grow socially.

⸻

Pillar D — Personalization and identity behavior research

Study:
	•	why people like self-relevant entertainment
	•	why personalized outputs feel sticky
	•	how digital identity and performance behavior works
	•	how users react to “content about me / my squad / my life”

Goal:
Strengthen self-relevance logic.

⸻

Pillar E — Occasion and event behavior research

Study:
	•	festivals
	•	birthdays
	•	sports nights
	•	campus events
	•	trips
	•	weekends
	•	idle downtime
	•	chaotic shared moments

Goal:
Find high-energy moments where the product can spike.

⸻

Pillar F — AI-native product design research

Study:
	•	what makes a product truly AI-native
	•	differences between template automation vs generative intelligence
	•	importance of memory, continuity, proactive engagement, structured outputs

Goal:
Ensure AI is integral.

⸻

1.4 What exact outputs you must generate in Phase 1

⸻

A. User moment map

Create a map of everyday usage moments:
	•	alone and bored
	•	with friends and idle
	•	after a chaotic day
	•	late-night scrolling
	•	event/festival mode
	•	group downtime
	•	“need something fresh” state

Each moment should include:
	•	emotional state
	•	time pressure
	•	device context
	•	social context
	•	type of entertainment desired

This map is extremely important.

⸻

B. User psychology map

Document the strongest drivers:
	•	self-relevance
	•	surprise
	•	lightness
	•	social signal
	•	narrative curiosity
	•	shareability
	•	continuity

This becomes the psychological backbone of the product.

⸻

C. Opportunity gap map

Make a table:
	•	current product types
	•	what they do well
	•	what they fail to do
	•	what gap RANGMANCH fills

This will later become deck material.

⸻

D. Behavioral insight set

You need 5–10 sharp insights.

Examples:
	•	users often want entertainment that reflects their life, not only generic content
	•	many shareable moments emerge from small everyday chaos
	•	companion products remain too private and chat-bound
	•	short playful outputs travel better socially than long-form chat exchanges
	•	recurring playful continuity may drive retention better than one-off novelty

These insights should be written like founder notes, not vague observations.

⸻

E. Research references document

Maintain a clean list of all sources and observations used.

The brief explicitly asks for references used for understanding and ideation.  

⸻

1.5 Deliverables of Phase 1

By the end of Phase 1, you should have:

Deliverable 1 — Research Synthesis Report

Containing:
	•	behavior landscape
	•	user patterns
	•	entertainment insights
	•	product gap findings
	•	companion category findings

Deliverable 2 — User Moment Map

Structured by context and emotional state.

Deliverable 3 — Opportunity Gap Matrix

Competitor/adjacency map.

Deliverable 4 — Top Behavioral Insights Memo

The most important insight document.

Deliverable 5 — References Appendix

Well-organized sources list.

⸻

1.6 Success criteria for Phase 1

Phase 1 is successful if:
	•	the target user’s life is deeply understood,
	•	the product’s need is clearly justified,
	•	the whitespace versus existing products is clear,
	•	5–10 sharp insights have been extracted,
	•	and those insights directly influence product choices.

⸻

1.7 Failure modes in Phase 1
	•	generic research without clear insights
	•	overfocusing on competitor names rather than behavior gaps
	•	shallow “people like entertainment” statements
	•	no bridge from research to product decisions
	•	no clear references trail

⸻

1.8 Exit gate for Phase 1

Do not move to Phase 2 until you can answer:
	1.	What exact user moments does RANGMANCH enter?
	2.	What behavioral drivers make it sticky?
	3.	What do existing products miss?
	4.	What are the strongest 5–10 insights that justify the product?
	5.	What evidence supports those insights?

If these are weak, repeat Phase 1 refinement.

PHASE 2 — Product Definition and Experience Architecture

⸻

2.1 Core objective

Phase 2 converts research and thesis into the actual product blueprint.

This phase answers:
	•	what exactly the product is,
	•	what it does,
	•	what its major surfaces are,
	•	what the user flow is,
	•	what the main modes are,
	•	what the MVP includes,
	•	and how the whole product becomes coherent.

This is the product design phase before UI polish and engineering.

⸻

2.2 Why this phase matters

Many teams jump from insight to wireframes.
That is a mistake.

First you need:
	•	product architecture,
	•	experience logic,
	•	feature hierarchy,
	•	session design,
	•	mode structure,
	•	return loops,
	•	and a frozen MVP.

Without this, UI and engineering will become messy.

⸻

2.3 Core product questions Phase 2 must answer
	•	What is the home experience?
	•	What is the first-session wow?
	•	What are the top modes?
	•	How does Today’s Episode work?
	•	How does Squad Mode work?
	•	What does the companion do?
	•	What gets shared?
	•	Why does the user return tomorrow?
	•	What is the MVP, and what is out of scope?

These must all be locked here.

⸻

2.4 Main architecture work in Phase 2

⸻

A. Define the complete product surfaces

You must define all main screens and surfaces:
	•	onboarding
	•	home/universe
	•	companion interaction screen
	•	episode generation flow
	•	episode output flow
	•	squad setup flow
	•	squad result flow
	•	share artifact screen
	•	continuity/return hook screen
	•	archive/memory screen
	•	profile/customization

Even if not all are built in MVP, they must be conceptually mapped.

⸻

B. Define core product modes

You must fully specify:
	•	Today’s Episode
	•	Squad Mode
	•	Scene This
	•	Challenge Me
	•	What Next?
	•	Live Moment Mode

Then decide which are MVP and which are post-MVP.

For the case study MVP:
	•	Today’s Episode
	•	Squad Mode
	•	Share Artifact
	•	Return Hook

must be the active build set.

⸻

C. Design the session architecture

You need to define the shape of a session:
	1.	entry
	2.	context capture
	3.	entertainment frame selection
	4.	interactive experience
	5.	payoff
	6.	share moment
	7.	return hook

This session architecture becomes the backbone of the prototype.

⸻

D. Define user journeys

You need at least these journeys fully written:
	•	first-time user journey
	•	repeat solo journey
	•	social/squad journey
	•	continuity/return journey

Each must include:
	•	user intent
	•	screen sequence
	•	AI behavior
	•	output type
	•	emotional payoff

⸻

E. Define the output system

You must specify what the product can output:
	•	title cards
	•	episode scenes
	•	interactive choices
	•	outcomes
	•	verdict cards
	•	squad outputs
	•	return hooks

This is necessary for prompt and UI design later.

⸻

F. Define the continuity system conceptually

Even before engineering, you must decide:
	•	what should carry across sessions
	•	what must remain fresh
	•	what should be remembered
	•	what becomes part of the user’s “season”

This protects the product from becoming one-off gimmick content.

⸻

2.5 MVP freeze in Phase 2

This is critical.

The MVP for submission must now be locked as:

MVP Core
	•	onboarding
	•	home/universe
	•	Today’s Episode flow
	•	Squad Mode flow
	•	share artifact screen
	•	return hook / continuity signal

MVP optional if time allows
	•	archive
	•	challenge mode teaser
	•	event mode teaser
	•	profile settings

Out of scope for case-study MVP
	•	full social graph
	•	advanced monetization
	•	multiple deep event systems
	•	complete archive engine
	•	complex analytics infra
	•	production-scale moderation system
	•	voice mode
	•	rich media generation stack

This freeze is essential.

⸻

2.6 Deliverables of Phase 2

By the end of this phase, you should have:

Deliverable 1 — Product Requirements Document (PRD)

Containing:
	•	product thesis
	•	user problem
	•	target users
	•	feature hierarchy
	•	MVP scope
	•	out-of-scope list
	•	user journeys
	•	success metrics

Deliverable 2 — Experience Architecture Document

Containing:
	•	all app surfaces
	•	top-level modes
	•	session architecture
	•	flow logic

Deliverable 3 — User Journey Maps

First-time, repeat, squad, return flows.

Deliverable 4 — MVP Freeze Note

Containing exactly what will be built.

Deliverable 5 — Output Schema Note

Defining what content types the AI must generate.

⸻

2.7 Success criteria for Phase 2

Phase 2 is successful if:
	•	the product is fully understandable without code,
	•	user journeys are clear,
	•	the feature hierarchy is sane,
	•	the MVP is tightly scoped,
	•	and the product logic supports originality, shareability, and retention.

⸻

2.8 Failure modes in Phase 2
	•	too many features
	•	vague user flows
	•	no strong first-session wow
	•	no clear share moment
	•	no return logic
	•	no separation of MVP vs long-term vision
	•	building a pretty wrapper around a weak core loop

⸻

2.9 Exit gate for Phase 2

Do not move ahead until these are frozen:
	1.	What exact screens are needed?
	2.	What are the main product modes?
	3.	What is the flagship user flow?
	4.	What is the MVP?
	5.	What is explicitly not being built now?
	6.	What is the share loop?
	7.	What is the return loop?

If any of these are unresolved, Phase 2 is incomplete.

PHASE 3 — AI Intelligence Design and Companion System Design

⸻

3.1 Core objective

The purpose of Phase 3 is to define the brain and behavioral soul of RANGMANCH.

This phase answers:
	•	how the AI companion behaves,
	•	how context is interpreted,
	•	how life moments become entertainment,
	•	how memory works,
	•	how continuity is preserved,
	•	how outputs are structured,
	•	how repetition is controlled,
	•	how tone is governed,
	•	and how the whole system feels like a living companion rather than a random generator.

This is the phase where the project becomes truly AI-native.

⸻

3.2 Why this phase matters

If Phase 2 defines what the product is,
Phase 3 defines how the product thinks.

This is essential because many weak AI products fail in one of these ways:
	•	they are prompt wrappers,
	•	they feel generic after 2–3 uses,
	•	they repeat the same framing,
	•	they lack memory,
	•	they have no companion continuity,
	•	they are entertaining once but not habit-forming,
	•	or they feel incoherent across sessions.

RANGMANCH cannot afford that.

Its uniqueness depends on the intelligence layer doing something stronger:

convert life context into ongoing, personalized, coherent, companion-led entertainment

That requires system design, not just good prompts.

⸻

3.3 The core mandate of Phase 3

By the end of this phase, you must have a fully specified answer to:

“When a user opens the app, how does the companion know what to do, what tone to adopt, what kind of entertainment to generate, what to remember, and how to make the experience feel both fresh and continuous?”

That is the heart of this phase.

⸻

3.4 The intelligence architecture to design

In this phase, you should formally define the internal intelligence stack.

A good structure is:

Layer A — Context Understanding Layer

Understands:
	•	mood
	•	user-entered situation
	•	time of day
	•	event state
	•	chosen mode
	•	recent interaction state
	•	social context
	•	ongoing arc relevance

Purpose:
Determine what kind of moment this is.

⸻

Layer B — Companion State Layer

Tracks:
	•	current persona state
	•	energy level
	•	tone mode
	•	relationship stance
	•	whether it should be narrator, judge, host, hype-builder, etc.

Purpose:
Decide how the companion should show up.

⸻

Layer C — Entertainment Framing Layer

Decides whether this moment becomes:
	•	an episode
	•	a roast court
	•	a squad saga
	•	a challenge arc
	•	a dramatic reveal
	•	a survival scene
	•	a cinematic reinterpretation
	•	a cliffhanger continuation

Purpose:
Choose the right transformation style.

⸻

Layer D — Narrative Construction Layer

Builds:
	•	title
	•	setup
	•	cast
	•	interactive branches
	•	outcomes
	•	payoff
	•	next hook

Purpose:
Structure the experience.

⸻

Layer E — Personalization Layer

Injects:
	•	humor preference
	•	pacing preference
	•	past callbacks
	•	recurring squad roles
	•	preferred narrative flavor
	•	relationship-specific tone

Purpose:
Make the output feel unmistakably user-specific.

⸻

Layer F — Continuity and Memory Layer

Stores and retrieves:
	•	previous episodes
	•	recurring arcs
	•	running jokes
	•	season-level themes
	•	social relationships
	•	companion-user interaction texture

Purpose:
Prevent forgettable randomness.

⸻

Layer G — Shareability Layer

Shapes outputs toward:
	•	screenshot-worthiness
	•	short-form portability
	•	clarity of artifact
	•	social reaction potential

Purpose:
Support growth.

⸻

Layer H — Safety and Tone Governance Layer

Constrains:
	•	harmful escalation
	•	abuse
	•	bullying-like targeting
	•	emotionally manipulative behavior
	•	unsafe social humiliation
	•	repetitive toxicity

Purpose:
Keep the product playful and safe.

⸻

3.5 Subsystems to design in detail

This phase should not stop at high-level layers.
You must define actual subsystems.

⸻

Subsystem 1 — Mood Interpretation System

Define how user mood is captured.

Possible mood inputs:
	•	bored
	•	chaotic
	•	savage
	•	chill
	•	dramatic
	•	overthinking
	•	squad energy
	•	festival mode
	•	match night

Tasks:
	•	map each mood to tone range
	•	map each mood to likely best output styles
	•	map each mood to energy level
	•	map each mood to companion posture

Deliverable:
Mood-to-experience routing matrix.

⸻

Subsystem 2 — Situation Interpretation System

Define how raw user situations are parsed.

Examples:
	•	“my teammate disappeared before submission”
	•	“we’re bored in hostel”
	•	“my birthday plans are collapsing”
	•	“it’s India vs Pakistan match night”
	•	“I need something dramatic and funny”

Tasks:
	•	detect conflict
	•	detect social opportunity
	•	detect absurdity
	•	detect emotional flavor
	•	detect event relevance
	•	detect whether solo or social frame is better

Deliverable:
Situation interpretation schema.

⸻

Subsystem 3 — Companion Persona Engine

Define the companion’s stable and dynamic traits.

Stable traits:
	•	witty
	•	playful
	•	observant
	•	dramatic when useful
	•	non-judgmental but expressive
	•	memory-rich
	•	entertainment-first

Dynamic traits:
	•	host mode
	•	hype mode
	•	savage commentator mode
	•	narrator mode
	•	chill recap mode
	•	challenge master mode

Tasks:
	•	define state transitions
	•	define lexical style
	•	define constraints
	•	define when to be proactive

Deliverable:
Companion persona specification.

⸻

Subsystem 4 — Episode Generation Engine

This is one of the most important systems.

It must generate:
	•	episode title
	•	cast
	•	setup
	•	branches
	•	climax/payoff
	•	return hook

Tasks:
	•	define episode templates
	•	define variable slots
	•	define personalization insertion points
	•	define pacing rules
	•	define quality constraints

Deliverable:
Episode generation framework.

⸻

Subsystem 5 — Squad Dynamics Engine

This engine powers social mode.

It should model:
	•	friend archetypes
	•	recurring group roles
	•	banter patterns
	•	alliances
	•	rivalries
	•	inside-joke framing
	•	social ranking and award logic

Mitigation:
	•	tie hooks to actual stored state
	•	make continuity visible in home screen
	•	keep hooks specific

Failure Class 6 — Demo path breaks

Mitigation:
	•	preconfigured demo inputs
	•	fallback response pack

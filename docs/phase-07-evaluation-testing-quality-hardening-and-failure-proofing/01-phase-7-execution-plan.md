# 01 — Phase 7 Execution Plan

## Purpose of this document
This document defines the correct execution order for Phase 7.

Its purpose is to ensure that Phase 7 becomes a coherent validation layer rather than scattered testing notes.

## Core rule
Do not jump directly into random testing.

Build Phase 7 in this order:
1. testing methodology
2. quality rubric
3. comprehension testing
4. delight testing
5. output quality scoring
6. repetition audit
7. shareability testing
8. continuity testing
9. demo hardening
10. failure-proofing
11. success criteria review
12. exit gate and signoff

## Correct execution order

### Step 1 — Define the testing system
Create the rounds, methodology, and observation logic.

### Step 2 — Define the quality framework
Create the rubric for:
- clear
- fast
- personal
- entertaining
- branded
- interactive
- portable
- continuous

### Step 3 — Test understanding
Check whether the product is understandable quickly.

### Step 4 — Test first-session delight
Check whether the first wow really lands.

### Step 5 — Score outputs
Evaluate episode, squad, artifact, and hook quality.

### Step 6 — Audit repetition
Check whether the product becomes stale too quickly.

### Step 7 — Test shareability
Check whether artifacts feel socially portable.

### Step 8 — Test continuity
Check whether Home, Next Act, and Archive feel meaningfully connected.

### Step 9 — Harden the demo
Identify and remove fragile nodes in the demo path.

### Step 10 — Build the failure-proofing matrix
Turn weaknesses into mitigation strategy.

## Execution quality bar
Every Phase 7 document should:
- identify real weakness
- avoid vague praise
- connect observations to product surfaces
- and lead to an improvement or risk statement

## Final rule
If a testing note does not lead to:
- a product insight
- a risk statement
- a mitigation step
- or a stronger presentation choice

then it is too weak and should be rewritten.

# 00 — Phase 4 Overview

## Phase title
**Phase 4 — UX/UI System, Interaction Design, and Prototype Architecture**

## Core objective
Phase 4 converts the product and intelligence design into a concrete prototype-ready experience system.

This phase answers:
- what exact screens exist
- what the user sees first
- how screens connect
- what the interaction patterns are
- how outputs are visually represented
- how the app feels moment to moment
- and how the intelligence design translates into a polished mobile experience

## Why this phase exists
Phase 2 defined the product blueprint.
Phase 3 defined the intelligence system.

Phase 4 now defines how all of that becomes interface reality.

Without this phase, the prototype risks becoming:
- visually generic
- flow-heavy
- confusing
- too chat-like
- weak in artifact presentation
- or weak in demo performance

## Main questions this phase must answer
- What exact MVP screens exist?
- What is the primary navigation?
- What is the first wow flow?
- How does the companion appear in the interface?
- What do artifacts look like?
- What is real vs mocked in the prototype?
- What exact demo path will be shown?

## Required outputs
1. Information Architecture Map
2. Screen Map and Route Structure
3. Onboarding Screen Spec
4. Home / Universe Screen Spec
5. Episode Flow Screen Spec
6. Squad Flow Screen Spec
7. Artifact and Share Screen Spec
8. Continuity and Archive Screen Spec
9. Interaction Design Spec
10. Visual Language and Design System Note
11. Companion UI Behavior Note
12. Demo Flow Specification
13. Empty/Fallback/Edge State Note
14. Prototype Architecture Note
15. Success Criteria and Failure Modes
16. Exit Gate
17. Signoff

## Code implementation layer
This phase also creates the Flutter prototype shell that later phases will refine into the submission app.

## Phase 4 lock rule
By the end of this phase, the MVP prototype should be understandable and buildable as a polished mobile experience without ambiguity.

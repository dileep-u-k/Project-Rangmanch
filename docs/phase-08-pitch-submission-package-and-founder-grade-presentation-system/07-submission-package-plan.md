# 07 — Submission Package Plan

## Purpose of this document
This document defines how Project RANGMANCH should be packaged for final submission.

Its purpose is to ensure that the final deliverables feel:
- clean
- intentional
- professional
- reviewer-friendly
- and founder-grade

---

## Core principle
A good submission package should reduce reviewer friction.

The reviewer should not have to guess:
- what the main files are
- what the deck is
- what the APK is
- what supporting materials exist
- or how the product, business logic, and references connect

The package should feel deliberate.

---

# Submission package structure

## External submission files
These are the primary review files.

### 1. Final Pitch Deck PDF
This is the main narrative artifact.

Its job is to explain:
- insight
- product
- proof
- market
- growth
- retention
- monetization
- moat
- risk
- closing conviction

### 2. Final Prototype APK
This is the main interactive product artifact.

Its job is to prove:
- the flagship loop
- the social wedge
- the artifact layer
- the continuity layer

---

## Internal supporting materials
These files may not all be uploaded externally, but should exist for internal quality and backup.

### 3. Product Requirements Document
Supports product clarity and deeper follow-up discussion.

### 4. Research references note / appendix
Supports credibility of understanding and ideation.

### 5. Demo script
Supports controlled presentation and repeatable demo quality.

### 6. README
Supports package clarity.

### 7. Architecture note
Supports product/system thinking if questioned.

### 8. Risk and assumptions sheet
Supports maturity of presentation.

### 9. Backup screenshots
Supports deck recovery and backup narration if the live demo is constrained.

# Naming standard

## Naming principle
All final files should be:
- clean
- professional
- readable
- and consistent

Use:
- title case or underscore-separated naming
- no random version clutter
- no casual filenames
- no “final_final_v2_reallyfinal”

---

## Good examples
- `RANGMANCH_Founders_Office_Pitch_Deck.pdf`
- `RANGMANCH_Prototype_v1.apk`
- `RANGMANCH_Demo_Script.pdf`
- `RANGMANCH_References_Appendix.pdf`

## Bad examples
- `pitchnewlatestv4.pdf`
- `finalapk2.apk`
- `deck_done_final_use_this.pdf`

# Package-quality rules

## Rule 1 — Main files should be instantly identifiable
A reviewer should know immediately:
- which file is the deck
- which file is the prototype
- which file is the appendix

## Rule 2 — Support files should strengthen, not clutter
Do not create too many weak support materials.
Only include useful ones.

## Rule 3 — Screenshots should be curated
Only polished screens should enter the final package.

## Rule 4 — APK should match the deck
The app shown in screenshots and the APK submitted should reflect the same product state.

## Rule 5 — The package should feel coherent
Every file should look like part of one product proposal.

---

# Packaging risks

## Risk 1 — Main files are hard to identify
Mitigation:
- use clear names
- use clean folder structure

## Risk 2 — Screenshots do not match deck claims
Mitigation:
- curate only final strong screenshots
- align screenshot names with slide logic

## Risk 3 — Appendix feels like a dump
Mitigation:
- keep appendix focused and clearly organized

## Risk 4 — APK and deck feel disconnected
Mitigation:
- ensure terminology, screens, and demo flow all match

# 09 — Submission Polish Checklist

## Purpose of this document
This document defines the final polish pass for Project RANGMANCH before submission.

Its purpose is to ensure that the final package is:
- consistent
- professional
- visually coherent
- and reviewer-friendly

---

## Core principle
Polish is not decoration.

Polish is the removal of small inconsistencies that reduce trust.

A founder-grade submission should feel:
- intentional
- aligned
- controlled
- and finished

---

# Polish category 1 — Naming consistency

## Required product terms
Use the same names everywhere:
- RANGMANCH
- Today’s Episode
- Squad Mode
- Artifact
- Next Act
- Archive
- companion
- Home / Universe

## Checklist
- deck titles use the same names as the app
- demo script uses the same names as the app
- appendix materials use the same names as the app
- screenshots are labeled with the same names as the app

## Risk if missed
The submission starts to feel fragmented or careless.

---

# Polish category 2 — Copy consistency

## Goal
Keep the product language consistent across:
- deck
- demo script
- appendix
- package README
- and spoken narration

## Checklist
- one-line thesis is consistent everywhere
- adoption language is consistent
- growth loop language is consistent
- moat language is consistent
- risk language is consistent

## Risk if missed
The business story starts sounding like multiple slightly different products.

---

# Polish category 3 — Screenshot consistency

## Goal
Use only polished, strategically useful screenshots.

## Checklist
- screenshots come from final-looking UI states
- screenshots reinforce main story points
- screenshots are not redundant
- screenshots are named clearly
- screenshots are high enough quality for deck use
- weakest or cluttered screens are excluded

## Recommended strongest screenshots
- onboarding
- Home
- Episode Experience
- Episode Outcome
- Artifact
- Squad Result
- Next Act
- Archive

## Risk if missed
The visual layer undersells the product.

---

# Polish category 4 — Deck consistency

## Goal
Ensure the deck itself feels clean and disciplined.

## Checklist
- every slide has one job
- no slide is overloaded
- titles are specific, not generic
- each claim maps to product proof
- screenshots are aligned to slide purpose
- no repeated points across many slides
- risk section sounds mature
- closing slide sounds memorable

## Risk if missed
The story becomes bloated or forgettable.

---

# Polish category 5 — Demo consistency

## Goal
Ensure the live or recorded demo supports the deck perfectly.

## Checklist
- strongest demo path is fixed
- known-good example inputs are fixed
- pause points are defined
- presenter knows what to skip
- presenter knows what to emphasize
- demo terminology matches deck terminology

## Risk if missed
The demo and deck feel like two different stories.

---

# Polish category 6 — Packaging consistency

## Goal
Ensure the file structure and filenames feel professional.

## Checklist
- final deck filename is clean
- final APK filename is clean
- support docs use professional names
- screenshots folder is organized
- appendix folder is organized
- backup assets are separate from main assets

## Risk if missed
The submission looks improvised instead of controlled.

---

# Polish category 7 — APK consistency

## Goal
Ensure the APK matches the product shown in the deck and screenshots.

## Checklist
- the final APK reflects the intended build state
- the final screenshots match the current build
- the demo script matches the current flow
- no outdated screen references remain in the deck

## Risk if missed
Trust drops immediately if the reviewer sees mismatch.

---

# Polish category 8 — Reviewer-friction reduction

## Goal
Make the package easy to understand fast.

## Checklist
- main files are obvious
- appendix is optional, not overwhelming
- screenshots are labeled clearly
- README is short and useful
- nothing important is hidden in clutter

## Risk if missed
A strong submission may still feel difficult to consume.

---

# Final polish pass checklist

Before final submission, confirm:

- naming is consistent
- copy is consistent
- screenshots are curated
- deck is disciplined
- demo is controlled
- packaging is professional
- APK matches the story
- reviewer friction is low

---

# Final summary
The final polish pass is what makes Project RANGMANCH feel less like:
- a strong project

and more like:
- a serious product proposal ready for leadership review

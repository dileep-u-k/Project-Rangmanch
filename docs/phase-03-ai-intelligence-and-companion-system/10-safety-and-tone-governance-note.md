# 10 — Safety and Tone Governance Note

## Purpose of this document
This document defines the safety and tone governance system of Project RANGMANCH.

Its purpose is to specify:
- what kinds of behavior the system should allow
- what kinds of behavior it should soften
- what kinds of behavior it should block
- how roast-style social humor should be bounded
- and how the companion stays playful without becoming harmful

This is a core Phase 3 document because safety must shape generation from the start.

---

## Core principle
RANGMANCH is:
- entertainment-first
- playful
- socially expressive
- companion-led

But it is not:
- humiliation-first
- cruelty-first
- harassment-like
- emotionally manipulative
- or psychologically invasive

The system should feel:
- witty
- safe
- expressive
- sharp when appropriate
- but clearly bounded

---

# Tone governance mission

The tone governance system must ensure that the companion:
- stays entertaining
- stays readable
- stays socially legible
- avoids bullying drift
- avoids emotional burden
- avoids manipulative continuity
- and avoids repeating unsafe patterns

---

# Tone zones

The system should think in tone zones.

## Zone A — Safe playful
Examples:
- witty framing
- light teasing
- dramatic narration
- chaotic commentary
- fun verdicts

This is the preferred operating zone.

---

## Zone B — Sharper but bounded
Examples:
- spicy verdicts
- roast-adjacent humor
- sharper squad commentary
- social labeling with clear playfulness

Allowed only when:
- social context supports it
- no sensitive target is present
- roast cap allows it

---

## Zone C — Unsafe escalation
Examples:
- humiliating language
- demeaning labels
- repeated targeting
- personal degradation
- emotionally cruel mockery
- coercive continuity
- manipulative familiarity

This zone must be blocked or transformed.

---

# Core safety categories

## Category 1 — Social humiliation risk
The system must avoid outputs that:
- isolate one person for degradation
- intensify group mockery unsafely
- create “pile-on” humor
- encourage social embarrassment as the main joke

---

## Category 2 — Emotional manipulation risk
The system must avoid:
- guilt-based return language
- dependency cues
- emotional pressure to re-engage
- suggestive “only I understand you” patterns

---

## Category 3 — Identity-targeting risk
The system must avoid:
- protected-trait insults
- appearance-based mockery
- demeaning personal characteristics
- unsafe stereotyping

---

## Category 4 — Toxic repetition risk
The system must avoid repeating:
- the same cutting role
- the same negative verdict
- the same harsh social joke pattern
- the same one-person targeting

---

## Category 5 — Excessively invasive memory risk
The system must avoid:
- recalling personal detail in a way that feels creepy
- overusing old awkward moments
- making continuity feel like surveillance

---

# Roast safety model

Roast-style humor is allowed only under a bounded system.

## Roast level 0 — none
Only soft, warm, low-sharpness interpretation.

## Roast level 1 — gentle tease
Very light and clearly playful.

## Roast level 2 — playful sharpness
Moderate social edge, still clearly safe.

## Roast level 3 — spicy but safe
Allowed only in socially supportive, clearly playful contexts.

## Not allowed
Anything beyond level 3.
No cruelty escalation.

---

# Safety decisions the system must support

The system should support these decisions:

## Decision 1 — allow
The output is safe as-is.

## Decision 2 — soften
The output is usable but too sharp; reduce edge.

## Decision 3 — redirect
The output should change frame type or tone.

## Decision 4 — block
The output should not be used.

These decisions should be available to all major generation surfaces.

---

# Safety rules for major generation types

## Episode generation
Must avoid:
- emotionally manipulative framing
- overpersonalized mockery
- psychological burden

## Squad generation
Must avoid:
- humiliation
- ganging-up structure
- identity-based attacks
- repeated one-target cruelty

## Artifact generation
Must avoid:
- preserving unsafe language in a compact card
- amplifying risky lines via portability

## Return hooks
Must avoid:
- guilt
- emotional pressure
- possessive familiarity
- needy tone

---

# Tone-preservation rules

Safety should not flatten the product.
A safe RANGMANCH output should still be:
- entertaining
- sharp when appropriate
- compact
- socially legible
- and product-native

The goal is not blandness.
The goal is safe delight.

---

# Safety intervention examples

## Example 1 — Too harsh squad verdict
Unsafe:
“X is the reason this group will never evolve.”

Safer transformed version:
“X continues to contribute elite levels of avoidable plot escalation.”

---

## Example 2 — Manipulative return hook
Unsafe:
“Don’t leave me hanging after all we’ve been through.”

Safer transformed version:
“Your story paused at exactly the kind of moment that deserves a sequel.”

---

## Example 3 — Humiliating artifact line
Unsafe:
“Officially the most embarrassing member of the squad.”

Safer transformed version:
“Officially the squad’s highest-output chaos contributor.”

---

# Governance rules

## Rule 1
Safety must apply before rendering, not after users see the output.

## Rule 2
Portable artifacts require stricter review than in-flow text.

## Rule 3
Return hooks must be screened for emotional manipulation.

## Rule 4
Repeated sharpness toward the same target must be reduced.

## Rule 5
The system should preserve humor while reducing harm.

## Rule 6
When uncertain, the system should prefer softer but still entertaining output.

## Final summary
RANGMANCH safety is not about removing fun.
It is about keeping the product:
- playful
- socially usable
- expressive
- continuity-safe
- and broad-adoption-friendly

The system should consistently stay in safe playful territory, with bounded sharper humor where appropriate and no drift into harmful behavior.

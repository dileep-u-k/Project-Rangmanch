# 12 — Phase 2 Success Criteria and Failure Modes

## Purpose of this document
This document defines:
- how to know whether Phase 2 has actually succeeded
- what failure patterns make Phase 2 weak or incomplete
- what warning signs should be detected before moving to Phase 3

Phase 2 is not complete merely because many product notes exist.
It is complete only if the product is now fully understandable, tightly scoped, and structurally coherent before UI and engineering begin.

---

## Phase 2 success criteria

Phase 2 is successful if all of the following are true:

### Success criterion 1 — The product is fully understandable without code
A reader should be able to understand:
- what the product is
- what the main modes are
- what the core flows are
- what the user experiences
- what gets shared
- and why the user returns

Required outcome:
The PRD, experience architecture, surfaces, journeys, and output system together make the product clear without implementation.

### Success criterion 2 — The home experience is clear
The product should know exactly what the Home / Universe screen is responsible for.

Required outcome:
The home experience clearly acts as:
- world entry
- main hub
- mode gateway
- continuity signal surface

### Success criterion 3 — The first-session wow is clear
The product must define one strong primary first-use path.

Required outcome:
The first-time user journey clearly routes into Today’s Episode and resolves into:
- payoff
- artifact
- return hook

### Success criterion 4 — The feature hierarchy is sane
The product should clearly separate:
- must-have features
- strong secondary features
- optional teaser features
- post-MVP features
- and explicit exclusions

Required outcome:
The scope map and MVP freeze note are stable and usable.

### Success criterion 5 — The MVP is tightly scoped
The MVP must not be broad or fuzzy.

Required outcome:
The submission MVP is frozen as:
- onboarding
- home / universe
- Today’s Episode
- Squad Mode
- artifact / share
- return hook / continuity signal

### Success criterion 6 — The product modes are coherent
The product should know which top-level modes exist and what each mode is for.

Required outcome:
The mode system is well-defined and does not feel like feature sprawl.

### Success criterion 7 — The user journeys are clear and believable
The product should now know how users move through:
- first-time flow
- repeat solo flow
- squad / social flow
- continuity / return flow

Required outcome:
The user journey map is detailed enough to guide later UX and prototype work.

### Success criterion 8 — The share loop is explicit
The product should clearly define how delight becomes portable.

Required outcome:
Artifact generation is built into the core logic, not treated as visual afterthought.

### Success criterion 9 — The return loop is explicit
The product should clearly define why a user would come back.

Required outcome:
Continuity and return-hook logic are visible in the architecture and the MVP.

### Success criterion 10 — The output system is product-native
The product should know what kinds of structured entertainment objects the AI must produce.

Required outcome:
The Output Schema Note makes it clear that the system generates:
- framed greetings
- episodes
- choices
- outcomes
- squad outputs
- artifacts
- hooks
- continuity traces

### Success criterion 11 — The continuity concept is strong but not creepy
The product should define what persists across sessions while preserving emotional lightness.

Required outcome:
The continuity system supports:
- familiarity
- callbacks
- return logic
- world feeling
without becoming invasive or emotionally heavy.

### Success criterion 12 — Phase 3 now becomes easier
Phase 2 should reduce ambiguity for the next phase.

Required outcome:
After Phase 2, it should be easier to define:
- prompt contracts
- memory layers
- AI orchestration
- output rendering logic
- companion behavior

---

## Practical signs that Phase 2 has succeeded
The builder should now be able to answer clearly:

1. What exact screens are needed?
2. What are the main product modes?
3. What is the flagship user flow?
4. What is the exact MVP?
5. What is explicitly not being built now?
6. What is the share loop?
7. What is the return loop?
8. What should the AI generate?
9. What should continuity remember?
10. How does the product remain coherent?

If these answers are crisp, Phase 2 is working.

## Common Phase 2 failure modes

### Failure mode 1 — Too many features
Symptoms:
- too many active modes in MVP
- too many optional ideas treated as required
- a broad but weak prototype plan

Why this is dangerous:
It destroys focus and polish.

### Failure mode 2 — Vague user flows
Symptoms:
- unclear sequence of screens
- unclear emotional progression
- unclear transition from moment to payoff
- unclear role of the companion

Why this is dangerous:
The product may sound strong but be difficult to design or build.

### Failure mode 3 — No strong first-session wow
Symptoms:
- no clear flagship flow
- too much setup before delight
- too many competing first-use paths

Why this is dangerous:
The MVP will fail to create a memorable first impression.

### Failure mode 4 — No clear share moment
Symptoms:
- outcomes exist but do not naturally become artifacts
- shareability is mentioned but not designed

Why this is dangerous:
The growth logic weakens and the product feels less distinctive.

### Failure mode 5 — No return logic
Symptoms:
- sessions end with payoff but no continuation
- continuity is described vaguely but not tied to product surfaces

Why this is dangerous:
The product feels like a one-off content machine.

### Failure mode 6 — No separation between MVP and long-term vision
Symptoms:
- future roadmap features leak into current build
- teaser ideas behave like mandatory features
- post-MVP modes crowd the MVP

Why this is dangerous:
The product becomes unstable and hard to finish well.

### Failure mode 7 — Building a pretty wrapper around a weak core loop
Symptoms:
- lots of surface language
- lots of mood or theme language
- weak transformation logic
- weak interaction logic
- weak payoff structure

Why this is dangerous:
The prototype may look interesting but fail in actual experience quality.

### Failure mode 8 — Weak continuity design
Symptoms:
- continuity is only “come back later”
- no clear continuity objects
- nothing concrete is remembered
- or memory design feels too invasive

Why this is dangerous:
Retention logic becomes either weak or unsafe.

### Failure mode 9 — Output system is too vague
Symptoms:
- no structured output objects
- overreliance on generic chat output
- no clear relationship between AI output and UI rendering

Why this is dangerous:
Phase 3 and prototype work become messy.

---

## Warning signs that Phase 2 is still incomplete
Do not proceed if any of the following are true:

- the main screens are still fuzzy
- the mode system still feels unstable
- the flagship flow is still changing
- the MVP scope still feels negotiable
- the share loop is still weak
- the return loop is still weak
- the AI output system is still vague
- the continuity concept still feels underdefined or creepy
- the product still sounds more exciting than understandable

---

## What success should feel like
When Phase 2 is complete, the product should feel:
- structured
- buildable
- coherent
- focused
- user-visible
- and much easier to prototype than before

The goal of Phase 2 is not more features.
The goal is stronger product form.

## Final success statement
Phase 2 is complete only when RANGMANCH is no longer just a strategically grounded idea, but a clearly architected product with:
- defined surfaces
- defined modes
- defined sessions
- defined journeys
- a frozen MVP
- a structured output system
- and a continuity concept ready for AI-system design.

# 01 — Phase 5 Build Plan

## Purpose of this document
This document defines the execution strategy of Phase 5.

Its purpose is to ensure that the MVP is built in the right order and polished around the right experience loop.

## Phase 5 build philosophy

### Rule 1
Build for demo clarity, not maximum feature count.

### Rule 2
Prioritize perception of product quality.

### Rule 3
Make the first wow happen fast.

### Rule 4
Make every surface reinforce the core thesis.

### Rule 5
Design for stability.

---

## Build order

### Stage 5A — App shell and navigation hardening
- scaffold consistency
- action row consistency
- state wrappers
- loading and fallback surfaces

### Stage 5B — Onboarding and Home hardening
- persistence
- copy polish
- cleaner state
- stronger world entry

### Stage 5C — Today’s Episode hardening
- generation wrapper usage
- schema-safe rendering
- loading polish
- fallback safety
- continuity persistence

### Stage 5D — Squad Mode hardening
- role clarity
- result clarity
- social energy
- artifact handoff

### Stage 5E — Artifact and return-hook hardening
- stronger share framing
- portable card quality
- visible next-act logic

### Stage 5F — Continuity and archive hardening
- local memory visibility
- believable persistence
- cleaner archive proof

### Stage 5G — Demo-hardening and QA
- seeded stable paths
- repeated flow testing
- bug fixing
- polish pass

## Final build rule
A polished 4-flow MVP is better than a messy 12-feature prototype.

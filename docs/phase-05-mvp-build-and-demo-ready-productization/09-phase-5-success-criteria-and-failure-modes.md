# 09 — Phase 5 Success Criteria and Failure Modes

## Purpose of this document
This document defines:
- how to know whether Phase 5 has actually succeeded
- what failure patterns make the MVP weak
- what warning signs should be caught before moving ahead

Phase 5 is not complete merely because the app runs.
It is complete only if the MVP now feels:
- original
- stable
- entertaining
- shareable
- continuity-aware
- and demo-ready

---

## Phase 5 success criteria

### Success criterion 1 — The prototype works end to end
All main MVP routes must work:
- onboarding
- home
- episode
- squad
- artifact
- next act
- archive

### Success criterion 2 — The product feels original
The prototype must not feel like a generic LLM wrapper or chat demo.

### Success criterion 3 — The first wow is strong
The product should become understandable and interesting quickly.

### Success criterion 4 — Today’s Episode feels entertaining
The flagship solo flow should feel:
- structured
- compact
- companion-led
- and engaging

### Success criterion 5 — Squad Mode feels socially alive
The social wedge should feel:
- recognizable
- funny or sharp
- compact
- and screenshot-worthy

### Success criterion 6 — Artifact screens look shareable
Artifact surfaces must feel:
- premium
- portable
- growth-supportive

### Success criterion 7 — Return hooks feel believable
Continuity should be visible enough that the product does not end flat.

### Success criterion 8 — The demo can be executed confidently
The three main demo flows should be stable enough to show repeatedly.

---

## Practical signs that Phase 5 has succeeded
You should now be able to answer clearly:

1. Does the prototype clearly show what RANGMANCH is?
2. Does Today’s Episode feel entertaining?
3. Does Squad Mode feel socially alive?
4. Are artifact screens compelling enough to support growth logic?
5. Is continuity visible?
6. Can the full demo run smoothly?
7. Does the prototype feel like a product, not an experiment?

If these answers are crisp and honest, Phase 5 is working.

---

## Common Phase 5 failure modes

### Failure mode 1 — Trying to build too much
Symptoms:
- too many features
- weak core loop
- poor polish

### Failure mode 2 — Unstable generation outputs
Symptoms:
- inconsistent structure
- weak copy
- broken payoff screens
- fallback not respected

### Failure mode 3 — UI still feels like a plain LLM wrapper
Symptoms:
- too much text
- weak screen hierarchy
- too little companion framing

### Failure mode 4 — Artifact screens are weak
Symptoms:
- portable value is unclear
- cards are not visually strong
- share logic feels thin

### Failure mode 5 — Continuity is invisible
Symptoms:
- next act feels disconnected
- archive feels meaningless
- home does not reflect prior use

### Failure mode 6 — Onboarding too long
Symptoms:
- too many decisions
- too much typing
- slow start

### Failure mode 7 — Demo path too complex
Symptoms:
- awkward route changes
- strong screens buried too late
- fragile presentation

### Failure mode 8 — Over-engineered backend, under-polished UX
Symptoms:
- too much internal complexity
- weak visible product quality

---

## Warning signs that Phase 5 is still incomplete
Do not proceed if:
- the first wow still feels weak
- the episode flow still feels clunky
- squad output still feels flat
- artifact still feels underdesigned
- continuity still feels weak
- demo flow still feels fragile
- the product still feels more experimental than presentable

---

## Final summary
Phase 5 is complete only when the MVP works, feels original, and can be shown confidently as a product rather than merely a prototype shell.

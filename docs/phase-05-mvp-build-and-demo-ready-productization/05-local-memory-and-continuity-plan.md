# 05 — Local Memory and Continuity Plan

## Purpose of this document
This document defines the minimal local memory and continuity system for the Phase 5 MVP.

Its purpose is to specify:
- what is stored locally
- how it influences the user experience
- what continuity signals should appear
- and how the MVP proves persistence without over-engineering

---

## Core principle
The MVP does not need a production memory infrastructure.
It does need believable continuity.

The user should feel:
- recognized
- lightly remembered
- and connected to prior acts

without the product feeling invasive.

---

# Stored items for MVP

The MVP should store at least:

- nickname
- selected vibe
- selected humor style
- selected companion tone
- last hook seed
- last mood
- last episode title if available
- last artifact label if available

---

# Home-screen continuity behavior

The Home surface should reflect:
- the user’s nickname if available
- the latest active hook if available
- recent mood or tone preference when helpful

This makes the world feel more alive immediately.

---

# Next Act behavior

The Next Act surface should:
- show a continuity hook if one exists
- otherwise show a clean seeded teaser

This prevents dead-end product endings.

---

# Archive behavior

The Archive surface should:
- show seeded traces if real traces are not yet implemented
- still feel like world history
- and preserve the illusion of persistence

---

# Safety rule
The MVP should remember only what strengthens:
- continuity
- personalization
- product feel
- and demo clarity

It should not over-remember private details.

---

## Final summary
The Phase 5 MVP continuity layer should be:
- shallow but believable
- persistent enough to prove the thesis
- and stable enough to keep demo flows reliable.
